    id         INTEGER PRIMARY KEY AUTOINCREMENT,

## 1. Проектиране на Бази Данни (Database Design)

### Какво е проектиране на бази данни?

**Проектирането** е процесът на решение **кои таблици да съществуват**, **кои колони да имат** и **как да са свързани помежду си**, преди да започнем да пишем SQL.

Доброто проектиране следва няколко основни принципа:

| Принцип | Описание |
|---------|----------|
| **Нормализация** | Схемата да не повтаря едни и същи данни ненужно |
| **Атомарност** | Всяка колона да съдържа само **едно** парче информация |
| **Интегритет** | Данните да са валидни и последователни по всяко време |
| **Минимална излишност** | Никоя информация да не се съхранява на две места |

### Пример: Лошо vs Добро проектиране

**Лошо (само 1 таблица):**
```
| book_title          | author_name    | author_country | reader_name | rented_on  |
|---------------------|----------------|----------------|-------------|------------|
| Под Игото           | Иван Вазов     | България       | Мартина     | 2024-01-10 |
| Под Игото           | Иван Вазов     | България       | Галина      | 2024-04-01 |
```
→ Данните за автора се **повтарят** за всяка книга, за всяко наемане. Промяна на страна налага обновяване на стотици редове.

**Добро (нормализирано, 4 таблици):**
```
authors  →  books  →  rentals  ←  readers
```
→ Данните за автора стоят **само на едно място** в `authors`. Книгите само сочат към тях с `author_id`.

## 2. Релации между таблиците (Table Relations)

Таблиците се свързват чрез **Първичен ключ (Primary Key)** и **Външен ключ (Foreign Key)**.

### Видове релации

#### 1:1 (Едно към Едно)
Всеки ред в таблица А съответства точно на **един** ред в таблица Б.

```
readers       reader_profiles
──────────    ───────────────
id  name      reader_id  bio
1   Мартина   1          "Обича класика"
2   Стефан    2          "Предпочита sci-fi"
```

#### 1:N (Едно към Много)  ← Най-честата релация
Един ред в таблица А може да е свързан с **много** редове в таблица Б.

```
authors(1)  →  books(N)
──────────────────────────────────
Иван Вазов  → "Под Игото", "Нова Земя"
Харуки Мур. → "Норвежка Гора", "Кафка на Брега", "Усамотена Планета"
```

```sql
-- author_id в books е FOREIGN KEY към id в authors (1:N)
CREATE TABLE books (
    ...
    author_id INTEGER NOT NULL,
    FOREIGN KEY (author_id) REFERENCES authors(id)
);
```

#### M:N (Много към Много)
Много редове от таблица А могат да са свързани с много редове от таблица Б.  
Реализира се с **свързваща (junction/bridge) таблица**.

```
books  ←  rentals  →  readers
─────────────────────────────
Книга 1 → наета от → Читател 1, Читател 3
Книга 7 → наета от → Читател 2, Читател 3
Читател 1 → наел → Книга 1, Книга 5
```

`rentals` е именно такава свързваща таблица — съдържа `book_id` и `reader_id`.

## 3. E/R Диаграми (Entity-Relationship Diagrams)

**E/R диаграмата** е визуалното представяне на схемата на базата данни. Показва **entity-тата (таблиците)**, техните **атрибути (колоните)** и **релациите** между тях.

### Нотация (Crow's Foot)

```
||──o{     =  Задължително едно към нула-или-много
||──||     =  Задължително едно към точно едно
o|──o{     =  Нула-или-едно към нула-или-много
```

### E/R Диаграма на нашата библиотека

```
┌──────────────────┐          ┌──────────────────────┐
│     AUTHORS      │          │        BOOKS         │
├──────────────────┤          ├──────────────────────┤
│ PK id            │ 1      N │ PK id                │
│    first_name    │──────────│ FK author_id         │
│    last_name     │          │    title             │
│    country       │          │    genre             │
└──────────────────┘          │    year              │
                              └──────────┬───────────┘
                                         │ 1
                                         │
                                         │ N
                              ┌──────────┴───────────┐
                              │       RENTALS        │
                              ├──────────────────────┤
                              │ PK id                │
                              │ FK book_id           │
                              │ FK reader_id         │
                              │    rented_on         │
                              │    returned_on       │
                              └──────────┬───────────┘
                                         │ N
                                         │
                                         │ 1
                              ┌──────────┴───────────┐
                              │       READERS        │
                              ├──────────────────────┤
                              │ PK id                │
                              │    first_name        │
                              │    last_name         │
                              │    email             │
                              │    joined            │
                              └──────────────────────┘
```

### Как да четем диаграмата:
- Един **автор** може да има **много книги** (1:N)
- Една **книга** може да бъде наета **много пъти** (1:N)
- Един **читател** може да е направил **много наемания** (1:N)
- **Книги** и **читатели** са в **M:N** релация чрез `rentals`

## 4. JOIN — Свързване на данни от множество таблици

`JOIN` е механизмът, чрез който SQL обединява редове от различни таблици **въз основа на съвпадаща стойност**.

### Синтаксис

```sql
SELECT колони
FROM таблица_1
    JOIN таблица_2 ON таблица_1.ключ = таблица_2.ключ;
```

### Видове JOIN

#### INNER JOIN (или просто JOIN)
Връща само редовете, при които има **съвпадение и в двете таблици**.

```
Таблица A            Таблица B           INNER JOIN резултат
──────────────       ──────────────      ───────────────────
id │ name            id │ book_id        name │ book_title
1  │ Мартина         1  │ 1   ◄──┐       Мартина │ Под Игото
2  │ Стефан          2  │ 3   ◄──┤       Стефан  │ Бай Ганьо
3  │ Галина          3  │ 7       │       Галина  │ Гордост...
4  │ Борислав        4  │ 5       └──── само съвпаденията!
5  │ Цветелина       ...                 ...
```

```sql
-- Пример: Всяка книга с нейния автор
SELECT b.title, a.first_name, a.last_name, a.country
FROM books AS b
    INNER JOIN authors AS a ON b.author_id = a.id;
```

**Резултат:**
```
title                   | first_name | last_name     | country
------------------------|------------|---------------|----------------
Под Игото               | Иван       | Вазов         | България
Нова Земя               | Иван       | Вазов         | България
Бай Ганьо               | Алеко      | Константинов  | България
Вишнева Градина         | Антон      | Чехов         | Русия
Гордост и Предразсъдък  | Джейн      | Остин         | Великобритания
Норвежка Гора           | Харуки     | Мураками      | Япония
```

---

#### LEFT JOIN (LEFT OUTER JOIN)
Връща **всички** редове от **лявата** таблица + съвпадащите от дясната. Ако няма съвпадение, дясната страна е `NULL`.

```sql
-- Всички автори — дори тези без книги в системата
SELECT a.first_name, a.last_name, b.title
FROM authors AS a
    LEFT JOIN books AS b ON a.id = b.author_id;
```

**Разлика:** Ако имахме автор без въведени книги, той пак щеше да се появи — с `NULL` за `title`.

---

#### RIGHT JOIN (RIGHT OUTER JOIN)
Огледален образ на LEFT JOIN — всички редове от **дясната** таблица.

> **Бележка за SQLite:** SQLite не поддържа RIGHT JOIN директно — вместо това обърнете реда на таблиците и използвайте LEFT JOIN.

---

#### FULL OUTER JOIN
Всички редове от **двете** таблици. Ако няма съвпадение, съответната страна е `NULL`.

> **Бележка за SQLite:** Не се поддържа директно — симулира се с `LEFT JOIN UNION RIGHT JOIN`.

---

### JOINs с три или повече таблици

```sql
-- Наемания с имена на книги И читатели
SELECT
    r.rented_on,
    b.title         AS kniga,
    rd.first_name   AS chitatel,
    r.returned_on
FROM rentals AS r
    INNER JOIN books   AS b  ON r.book_id   = b.id
    INNER JOIN readers AS rd ON r.reader_id = rd.id
ORDER BY r.rented_on;
```

```sql
-- Пълна верига: наемания + книга + автор + читател
SELECT
    rd.first_name || ' ' || rd.last_name   AS chitatel,
    b.title                                 AS kniga,
    a.first_name  || ' ' || a.last_name    AS avtor,
    r.rented_on,
    CASE WHEN r.returned_on IS NULL THEN 'Не е върната'
         ELSE r.returned_on END             AS status
FROM rentals AS r
    INNER JOIN books   AS b  ON r.book_id   = b.id
    INNER JOIN authors AS a  ON b.author_id = a.id
    INNER JOIN readers AS rd ON r.reader_id = rd.id
ORDER BY r.rented_on;
```

---

### JOIN + GROUP BY (Агрегиране след свързване)

```sql
-- Колко книги има всеки автор?
SELECT
    a.first_name || ' ' || a.last_name AS avtor,
    COUNT(b.id)                         AS broi_knigi
FROM authors AS a
    LEFT JOIN books AS b ON a.id = b.author_id
GROUP BY a.id
ORDER BY broi_knigi DESC;
```

```sql
-- Кои читатели имат активни наемания (не са върнати)?
SELECT
    rd.first_name || ' ' || rd.last_name AS chitatel,
    COUNT(r.id)                           AS aktivni_naemaniya
FROM readers AS rd
    INNER JOIN rentals AS r ON rd.id = r.reader_id
WHERE r.returned_on IS NULL
GROUP BY rd.id
ORDER BY aktivni_naemaniya DESC;
```

## 5. Cascade операции

`CASCADE` инструктира базата данни **автоматично** да разпространи промяна в родителската таблица към свързаните дъщерни записи.

### Видове Cascade действия

| Действие | Описание |
|----------|----------|
| `ON DELETE CASCADE` | При изтриване на ред от родителя → изтрий и свързаните детски редове |
| `ON DELETE SET NULL` | При изтриване на ред от родителя → постави `NULL` в детските редове |
| `ON DELETE RESTRICT` | Забрани изтриването, ако има свързани детски редове (по подразбиране) |
| `ON UPDATE CASCADE` | При промяна на PK в родителя → обнови FK в детските редове |

### Пример: ON DELETE CASCADE

```sql
-- Нашата схема вече има CASCADE:
-- books.author_id → REFERENCES authors(id) ON DELETE CASCADE

-- Проверка преди:
SELECT COUNT(*) FROM books WHERE author_id = 5;
-- Резултат: 3 книги (Мураками)

-- Изтриваме автора
DELETE FROM authors WHERE id = 5;

-- Проверка след:
SELECT COUNT(*) FROM books WHERE author_id = 5;
-- Резултат: 0 — книгите са АВТОМАТИЧНО изтрити!
```

> **Важно за SQLite:** `CASCADE` работи само ако е активиран с `PRAGMA foreign_keys = ON;` — изпълнете го преди тестовете!

```sql
PRAGMA foreign_keys = ON;
```

### Пример: ON DELETE SET NULL

```sql
-- Ако читател бъде изтрит, наеманията му остават но reader_id → NULL
-- (наличък запис за историята без конкретен читател)

DELETE FROM readers WHERE id = 1;

SELECT * FROM rentals WHERE reader_id IS NULL;
-- Ще покаже наеманията на Мартина с reader_id = NULL
```

### Кога да използваме кое?

| Ситуация | Препоръка |
|----------|-----------|
| Изтриване на автор → книгите му нямат смисъл без него | `ON DELETE CASCADE` |
| Изтриване на читател → историята на наеманията е важна | `ON DELETE SET NULL` |
| Изтриване на категория с продукти (не трябва да се допуска) | `ON DELETE RESTRICT` |

---

## Обобщение

| Концепция | Ключово Take-away |
|-----------|-------------------|
| **Database Design** | Нормализирай! Не повтаряй данни — използвай FK. |
| **1:N Релация** | Най-честата — Foreign Key от дъщерната таблица сочи към Primary Key на родителската. |
| **M:N Релация** | Нужна е **junction table** с два Foreign Key-а. |
| **E/R Диаграма** | Визуална карта на схемата — незаменима при проектиране. |
| **INNER JOIN** | Само съвпадащите редове от двете таблици. |
| **LEFT JOIN** | Всички от лявата + съвпадащите от дясната (NULL при липса). |
| **JOIN + GROUP BY** | Свърши, после агрегирай — мощна комбинация. |
| **CASCADE** | Автоматично управление на свързани редове при DELETE/UPDATE. |
